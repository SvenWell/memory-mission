# AGENTS.md — canonical agent instructions

This file is the canonical instruction set for any AI agent working on Memory Mission. Host-specific entry points (`CLAUDE.md` at the repo root, future `GEMINI.md` / `CODEX.md` / etc.) are one-line shims that import this file via `@docs/AGENTS.md`.

Why canonical lives in `docs/` rather than repo root: the repo-root `AGENTS.md` is auto-generated by the `claude-mem` session-context tool and gets overwritten every session. Keeping our instructions at `docs/AGENTS.md` is the stable pattern (see ADR-0008, planned).

---

## 30-second orientation

**What this is.** Memory Mission is a governed context engine for agents — Python infrastructure that turns a firm's scattered knowledge (email, meetings, memos, CRM) into a structured, auditable memory layer that AI agents can read and write safely. Multi-user, multi-plane, provenance-mandatory. Start with `docs/VISION.md` for the full pitch.

**Shipped state.** V1 complete as of commit `a88f09a`. 17 build steps + 6 polish moves. 643 tests, mypy strict clean on 66 source files. Branch `SvenWell/office-hours` is ahead of `main` by 38 commits.

**What you're here to do** (in priority order):
1. If the user invokes a skill, invoke the Skill tool FIRST — don't answer directly. See "Skill routing" below.
2. If the user asks a question about the codebase, read `docs/VISION.md` → `docs/ARCHITECTURE.md` → `docs/ABSTRACTIONS.md` in that order. Those three answer most architecture questions without reading source.
3. If the user wants to build something new, read the matching step section in `BUILD_LOG.md` for the nearest prior work, then check `docs/adr/` for load-bearing decisions that constrain the space.
4. If the user asks for a code change, write tests alongside. `make check` (ruff + format + mypy --strict + pytest) must pass before commit.

## Core design rules you must not break

These are load-bearing. Violating them silently makes the whole model unsound. Each has an ADR or a doc section justifying it.

1. **Two planes, one-way bridge** (ADR-0002). `personal/<emp>/` is private; `firm/` only receives promoted facts. Nothing writes to `firm/` without going through `create_proposal` → `review-proposals` → `promote()` with reviewer + rationale.
2. **LLM lives with the host agent.** No SDK imports (`anthropic`, `openai`, `google-generativeai`) anywhere in `src/`. Memory Mission ships prompts, schemas, ingest validators. The host agent runs the model.
3. **Provenance mandatory.** Every fact carries `source_closet` + `source_file`. Every promotion records `reviewer_id` + `rationale`. Every corroboration appends to `triple_sources`. Never drop or bypass.
4. **Observability scope required for promotion.** `create_proposal` / `promote` / `reject` / `reopen` raise without an active `observability_scope()`. Audit trail is structural, not optional.
5. **Deterministic primitives, human judgment.** Mechanical steps (corroborate, coherence check, federated detect, merge entities) are pure Python + SQL. Editorial acts (approve, reject, tier changes, identity merges) always route through a human reviewer.
6. **Per-firm isolation.** One firm = one directory + one SQLite KG + one ProposalStore + one observability log. No cross-firm queries.
7. **Obsidian-compatible on-disk format.** Pages are markdown with YAML frontmatter. Anything you add to `PageFrontmatter` must survive Obsidian's parser and round-trip through `parse_page` / `render_page`.
8. **Files-first.** Filesystem is the source of truth for pages. KG and observability log are derived indexes that can be rebuilt from the promotion event history.

When in doubt, read the ADR for the relevant decision (`docs/adr/NNNN-*.md`).

## Skill routing

When the user's request matches an available skill, ALWAYS invoke it using the Skill tool as your FIRST action. Do NOT answer directly, do NOT use other tools first. The skill has specialized workflows that produce better results than ad-hoc answers.

Host-agent (gstack) skills:
- Product ideas, "is this worth building", brainstorming → invoke `office-hours`
- Bugs, errors, "why is this broken", 500 errors → invoke `investigate`
- Ship, deploy, push, create PR → invoke `ship`
- QA, test the site, find bugs → invoke `qa`
- Code review, check my diff → invoke `review`
- Update docs after shipping → invoke `document-release`
- Weekly retro → invoke `retro`
- Design system, brand → invoke `design-consultation`
- Visual audit, design polish → invoke `design-review`
- Architecture review → invoke `plan-eng-review`
- Save progress, checkpoint, resume → invoke `checkpoint`
- Code quality, health check → invoke `health`

## Memory Mission skills

Ship under `skills/<name>/SKILL.md` (registered in `skills/_index.md` + `skills/_manifest.jsonl`). When the user's request maps to one, invoke it:

- `backfill-gmail` / `backfill-granola` / `backfill-firm-artefacts` — pull source material into staging
- `extract-from-staging` — host LLM extracts typed facts from staged items
- `review-proposals` — merge-gate workflow for pending proposals (rationale required)
- `detect-firm-candidates` — admin: scan personal planes for cross-employee patterns
- `meeting-prep` — compile distilled context for a meeting / briefing task

Every skill is destinations-and-fences markdown with typed frontmatter. If you're writing a new skill, match the shape of existing ones.

## Build discipline

- **`make check`** runs ruff + format + mypy --strict + full pytest. Must pass before commit.
- **Commits are atomic per logical step.** Not "misc fixes"; not "updates." One commit = one shippable unit of work. See commit history on the branch for the expected style.
- **Every public surface gets tests.** No new Pydantic model, no new KG operation, no new skill without tests. Target lives next to the file it tests (`tests/test_<module>.py`).
- **No LLM calls in code.** See core rule 2. If you feel the need, you're in the wrong layer.
- **Tier + identity are load-bearing retrofits.** Don't reintroduce raw kebab-case entity names where a stable ID belongs. Don't reintroduce flat untyped triples where a tier belongs.
- **Branch workflow:** `SvenWell/office-hours` is the feature branch. Push commits to origin; the user opens PRs manually when ready.

## When you're stuck

- Look at `BUILD_LOG.md` for the nearest prior step that did something similar.
- Check `docs/adr/` for a load-bearing decision that might constrain the space.
- Grep `docs/ABSTRACTIONS.md` for the exact model / predicate / tier you're missing.
- Read the test file for the module you're changing — tests document the contract.

If after those four passes you're still stuck, surface it to the user with a specific question. Do not guess on architectural decisions.
